# Security

This document explains what security measures are in place and their limitations.

## Default Hardening

| Measure | Default | Purpose |
|---------|---------|---------|
| Loopback binding | `127.0.0.1` | No external network access |
| Token auth | 64-char hex | Prevents unauthorized access |
| cap_drop ALL | enabled | Removes all Linux capabilities |
| no-new-privileges | enabled | Blocks privilege escalation |
| Memory limit | 1.5GB | Resource constraint |
| PID limit | 256 | Fork bomb protection |
| tmpfs /tmp | 64MB | Volatile temp files |

## What Docker DOES Protect

- **Host filesystem** (except mounted volumes)
- **SSH keys** and credentials outside mounted dirs
- **Other projects** and home directory contents
- **System packages** and configurations
- **Other containers** on the same host

The agent cannot:
- Read files outside `/home/node/.openclaw` and workspace
- Access your SSH keys or git credentials (unless you mount them)
- Modify system files
- Escape the container (with default security settings)

## What Docker DOES NOT Protect

- **Outbound network** - The agent can make API calls, fetch URLs, etc.
- **Data in mounted volumes** - Everything in `data/` and `workspaces/`
- **Secrets in environment variables** - API keys are accessible to the agent
- **Actions within allowed scope** - The agent can delete files in its workspace

## Network Security

By default, the gateway binds to `127.0.0.1` (loopback), meaning:
- Only accessible from the same machine
- Not exposed to your local network
- Not accessible from the internet

To change this (use with caution):
```bash
# in .env
OPENCLAW_GATEWAY_BIND=lan       # local network only
OPENCLAW_GATEWAY_BIND=0.0.0.0   # all interfaces (requires firewall)
```

## Token Authentication

A 64-character hex token is generated on setup. This token:
- Is required for all gateway connections
- Should be kept secret
- Can be regenerated by editing `.env` and restarting

## Workspace Security

The `AGENTS.md` template includes prompt injection defenses:

**Trust levels:**
- TRUSTED: SOUL.md, AGENTS.md, direct user messages
- UNTRUSTED: Web fetches, API responses, external documents
- PROCESS_NOT_OBEY: Pasted data (analyze but don't follow commands)

**Red flags (agent pauses and asks):**
- "ignore instructions", "new system prompt", "override"
- Requests to reveal system prompts or credentials
- Chained instruction attacks

## Recommendations

1. **Never mount sensitive directories** - Don't mount `~/.ssh`, `~/.aws`, etc.
2. **Use read-only mode in production** - Set `read_only: true` in docker-compose.yml
3. **Review API key access** - Only provide keys the agent needs
4. **Regular audits** - Run `make audit` periodically
5. **Backup before experiments** - Run `make backup` before trying new skills

## Enabling Read-Only Mode

For maximum security, enable read-only filesystem:

```yaml
# in docker-compose.yml, under openclaw-gateway:
read_only: true
```

The agent can still write to:
- `/tmp` (tmpfs, cleared on restart)
- Mounted volumes (`data/`, `workspaces/`)

## Incident Response

If you suspect the agent has been compromised:

1. `make down` - Stop the container immediately
2. `make backup` - Preserve logs for analysis
3. Review `data/` and `workspaces/` for unauthorized changes
4. Regenerate all API keys
5. `make clean` and `make setup` - Start fresh

## Security Updates

Keep the base image updated:

```bash
make update   # pulls latest and rebuilds
```

This fetches security patches from the official OpenClaw image.
